## PART 1: CORE ARCHITECTURE â€” HOW THE HORIZONTAL SCROLL WORKS

### The Canvas
The entire About page is a single container laid out horizontally. Think of it as one enormously wide canvas â€” approximately 20x the viewport width. Each "slide" is a zone on this canvas, each zone being at least 100vw wide (many are wider). The viewport acts as a fixed frame. As the user scrolls down, the canvas translates left, revealing content from right to left.

### GSAP ScrollTrigger Setup
```js
import gsap from "gsap";
import { ScrollTrigger } from "gsap/ScrollTrigger";
gsap.registerPlugin(ScrollTrigger);

const container = document.querySelector(".slides-container");
const slides = gsap.utils.toArray(".slide");

// Calculate total width
const totalWidth = container.scrollWidth - window.innerWidth;

// Pin the container and translate it horizontally
const mainTimeline = gsap.to(container, {
  x: -totalWidth,
  ease: "none",
  scrollTrigger: {
    trigger: ".slides-wrapper",
    pin: true,
    scrub: 1, // 1 second of smoothing â€” critical for that buttery feel
    end: () => "+=" + totalWidth,
    invalidateOnRefresh: true,
  },
});
```

## PART 2: THE PARALLAX & TRANSITION SYSTEM â€” HOW ELEMENTS MOVE

This is the most important section. Every element on the page belongs to one of three parallax layers, and each layer moves at a different horizontal speed relative to the main scroll. This creates depth â€” like looking out of a car window where nearby objects move fast and distant objects move slow.

### Layer 1: Headlines (fastest)
Headlines translate horizontally at **1.15x** the base scroll speed. They enter from the right edge first and exit the left edge first. This makes headlines feel "close" to the viewer.

### Layer 2: Body Text / Statements (base speed)
Body text and statement paragraphs move at **1.0x** â€” they match the base horizontal scroll speed. This is the default layer.

### Layer 3: Images (slowest)
Images translate horizontally at **0.7â€“0.85x** the base scroll speed. They lag behind text. When a headline and an image are entering from the right at the same time, the headline arrives first and the image catches up. When they're both exiting left, the headline leaves first and the image lingers. This parallax gap is what creates the cinematic depth.

**Implementation approach:**
Each element within a slide has its own GSAP ScrollTrigger that adds a secondary horizontal transform on top of the main canvas translation:
```js
// Example: image parallax (moves slower = positive offset against the leftward scroll)
gsap.to(".slide-3 .parallax-image", {
  x: "15%", // positive x offsets the leftward movement, making image "lag"
  ease: "none",
  scrollTrigger: {
    trigger: ".slide-3",
    containerAnimation: mainTimeline, // tie to the horizontal scroll
    scrub: true,
    start: "left right", // when slide's left edge hits viewport right
    end: "right left", // when slide's right edge hits viewport left
  },
});

// Example: headline parallax (moves faster = negative offset)
gsap.to(".slide-3 .parallax-headline", {
  x: "-10%",
  ease: "none",
  scrollTrigger: {
    trigger: ".slide-3",
    containerAnimation: mainTimeline,
    scrub: true,
    start: "left right",
    end: "right left",
  },
});
```

### Transition Zones â€” NO HARD CUTS

This is critical. **There is NEVER a hard cut between slides.** Between every two slides there is a **300â€“500px overlap zone** where elements from both slides are visible simultaneously. The outgoing slide's elements are shrinking and fading in the left portion of the viewport while the incoming slide's elements are growing and appearing in the right portion.

**How to build overlap zones:**
Each slide is slightly wider than 100vw (add 200â€“400px of padding on the right side). Elements within a slide are positioned so they're visible even when the slide is partially off-screen. Images from the outgoing slide have `overflow: visible` or are positioned with negative margins so they bleed into the transition zone.

**What the viewer sees during EVERY transition (this repeats between each slide):**

- Frame 1 (settled): Slide A content fully visible, composed, readable. The user is dwelling here.
- Frame 2 (transition begins): Slide A content starts drifting left. The headline is the first thing to start leaving (fastest layer). The right edge of the viewport reveals pure black or a sliver of Slide B's first element peeking in from the right.
- Frame 3 (mid-transition): Slide A's image is now small, tucked into the left 20â€“30% of the viewport and still shrinking. Slide A's headline is gone (already off the left edge). Slide B's headline or label text is partially visible on the right 30â€“40%. Center of viewport is black negative space â€” breathing room.
- Frame 4 (late transition): Slide A's image is tiny in the far left corner or has faded to opacity 0. Slide B's headline is fully visible, centered or left-aligned. Slide B's image is entering from the right at ~40â€“60% of its final size. Still some black space between Slide B's text and image â€” they haven't composed yet.
- Frame 5 (new settled): Slide A is completely gone. Slide B's content is now fully composed â€” headline, text, images all in their final resting positions. The user dwells here.

**This 5-frame transition pattern is the fundamental rhythm of the entire page.** Every transition follows it. The timing varies (some transitions are faster, some slower) but the structure is always: settled â†’ text exits first â†’ image lingers â†’ overlap zone with black breathing room â†’ new content settles.

## PART 3: IMAGE BEHAVIOR SYSTEM

Images in this design are NOT static. They have complex scroll-driven behaviors. There are 4 distinct behaviors used across the page:

### Behavior A: Enter â†’ Grow â†’ Settle â†’ Shrink â†’ Exit
This is the default image behavior. Most images follow this lifecycle:

1. **Enter** from the right edge of the viewport at ~40â€“60% of their final size. The image is positioned off-screen right and begins moving leftward.
2. **Grow** as the image travels leftward, scaling up via `transform: scale()`. The scaling is scrubbed to scroll position so it's perfectly smooth.
3. **Settle** at full composed size when the slide is centered in the viewport. The image reaches scale(1.0) and its final position (e.g., right: 4%, top: 12%).
4. **Shrink** as the slide continues moving left, scaling back down. The image also drifts leftward within its slide, compressing toward the left edge.
5. **Exit** off the left edge at ~20â€“30% of their settled size. The image is now small in the far left corner before disappearing.

The key visual effect: the image appears to "breathe" â€” expanding on approach, holding at full size, then contracting on departure. This gives every image its own dramatic arc.

GSAP implementation:
```js
// ENTER + GROW (right edge â†’ centered)
gsap.fromTo(".slide-3 .hero-image", 
  { scale: 0.5, xPercent: 30, opacity: 0.7 },
  {
    scale: 1.0, xPercent: 0, opacity: 1,
    ease: "none",
    scrollTrigger: {
      trigger: ".slide-3",
      containerAnimation: mainTimeline,
      scrub: true,
      start: "left right",     // slide enters viewport
      end: "center center",    // slide is centered
    },
  }
);

// SHRINK + EXIT (centered â†’ left edge)
gsap.to(".slide-3 .hero-image", {
  scale: 0.3, xPercent: -30, opacity: 0.5,
  ease: "none",
  scrollTrigger: {
    trigger: ".slide-3",
    containerAnimation: mainTimeline,
    scrub: true,
    start: "center center",
    end: "right left",
  },
});
```

### Behavior B: Enter â†’ Scale to Full-Bleed â†’ Hold â†’ Shrink to Contained
Some images scale from a contained rectangle all the way to covering the ENTIRE viewport (100vw Ă— 100vh). This is used for major "hero moments" and is the most dramatic image behavior.

**How it looks frame by frame (from the reference, Screenshots 9â€“13 and 50â€“54):**

- Frame 1: Image is a rectangle occupying ~40% of the viewport width, positioned right side. Black background visible on all other sides.
- Frame 2: Image has grown to ~60% width. Top and bottom edges are closer to the viewport edges.
- Frame 3: Image is ~80% width. Only thin strips of black visible at the edges.
- Frame 4: Image is 100% width and 100% height. It IS the viewport. No black visible. Full-bleed.
- Frame 5: Image holds at full-bleed while text is overlaid on top of it.

**Then when exiting (Screenshots 58â€“61):**
- Frame 1: Image still full-bleed.
- Frame 2: Image begins shrinking. Width drops to ~70%. Black background reappears at the edges.
- Frame 3: Image now ~40% width, shifted to the LEFT side of the viewport. Right side of viewport is black.
- Frame 4: Image ~25% width, tucked into far left corner. New image entering from right side.
- Frame 5: Old image gone or tiny. New image settling into right side.

GSAP: Animate `width`, `height`, `top`, `left` (or use `scale` + `transformOrigin`) simultaneously, scrubbed to scroll. The image container must be `position: absolute` within the slide and the slide must be `overflow: hidden` (or `visible` during transitions if you want bleed).

### Behavior C: Pinned (counter-animated to stay fixed while canvas moves)
Used when text cycles through multiple states and the image needs to stay put. The image effectively becomes `position: fixed` relative to the viewport by counter-animating against the horizontal scroll.

**How it looks (from the reference, Screenshots 36â€“45):**
The golden foil portrait stays EXACTLY in the same screen position (right half) for the ENTIRE duration of the cycling text sequence (~10+ screenshots = a LOT of scroll distance). The text on the left changes, the user is scrolling, but the image doesn't budge. It only starts moving again when the cycling sequence ends and the slide begins its exit.

Implementation: Calculate exactly how many pixels the container will translate during the cycling phase, then apply a GSAP animation that moves the image the EXACT same distance in the OPPOSITE direction:
```js
const cycleScrollDistance = 250 * window.innerWidth / 100; // 250vw converted to px
gsap.to(".pinned-image", {
  x: cycleScrollDistance, // counter-animate
  ease: "none",
  scrollTrigger: {
    trigger: ".slide-8",
    containerAnimation: mainTimeline,
    scrub: true,
    start: "left+=100vw left", // start pinning after entrance
    end: "right-=100vw right", // stop pinning before exit
  },
});
```

### Behavior D: Dual-Image Bridge
Between major sections, two images sit side by side as a "visual bridge" â€” a purely cinematic pause with no text. One image belongs to the exiting section (left side, may be shrinking) and one belongs to the entering section (right side, may be growing).

**From the reference (Screenshots 48â€“50):**

- Right image: police car flood scene (beginning of "The Impact" section), taking up right 50% of viewport
- Small gap between them (~5% black)
- NO TEXT visible. Pure visual moment.
- Duration: brief â€” about 0.5x viewport width of scroll distance before the next phase begins

### Image Positioning Rules (apply to ALL images)
- Images are **NEVER** centered in the viewport. They are asymmetrically offset â€” anchored to one side or overlapping an edge.
- Vertical position varies between images: some sit high (top: 8%), some centered (top: 15%), some low (bottom: 10%). Alternate between these to create visual rhythm.
- Images overlap with text during transitions (text has higher z-index).
- Aspect ratios: use cinematic ratios â€” 16:9 for landscapes, 3:4 for tall portraits, 2.39:1 for ultra-wide hero shots.

## PART 4: TYPOGRAPHY SYSTEM

### Fonts
- **Display/Headlines:** A distinctive serif. Use Google Font `"DM Serif Display"` or `"Playfair Display"`. Weight: 400 for headlines, 700 for emphasized caps. These headlines should feel like film titles.
- **Body/Statements:** A clean, slightly geometric sans-serif. Use `"Syne"` or `"Outfit"` or `"Instrument Sans"` from Google Fonts. NOT Inter, NOT Roboto, NOT Arial, NOT system fonts.
- **Labels/Accent:** Same sans-serif as body, but: text-transform: uppercase, letter-spacing: 0.2em, color: #666666, font-size: clamp(11px, 1vw, 16px).

### Sizing (all use clamp for fluid scaling)
- **Massive display headlines** ("The Work Speaks", "Full Service", "The Impact", "Built Different"): `clamp(60px, 9vw, 160px)`. Line-height: 0.95. These are ENORMOUS. They dominate the viewport.
- **Section labels** ("What We Do", "One Team"): `clamp(11px, 1.2vw, 16px)`. Uppercase. Letter-spaced.
- **Statement paragraphs** (the keyword-emphasis blocks in Slides 6 and 7): `clamp(24px, 3.5vw, 52px)`. Line-height: 1.3. These are BIG â€” almost headline scale. They're meant to be read as singular visual statements, not body copy.
- **Cycling body lines** ("We conceptualize.", "Producers who've managed..."): `clamp(20px, 2.8vw, 40px)`. Line-height: 1.4.
- **Manifesto stacking lines** ("We conceptualize." etc.): `clamp(18px, 2.5vw, 36px)`. Line-height: 1.6.
- **Closing CTA**: `clamp(22px, 3vw, 42px)`. Line-height: 1.4.
- **Email / small text**: 14px, #666666, letter-spacing: 0.1em.
